# Codex Panes And Page Context

Telescope models agent UIs as docked panes. A pane can be placed on the `left`, `right`, or `bottom` of the browser surface and attached to the active page tab.

Browser tabs are also explicit runtime actions. `POST /v1/tabs` creates a tab state, makes it active, and queues an `open_tab` browser command; `POST /v1/tabs/{tab_id}/activate` queues an `activate_tab` browser command; `POST /v1/tabs/{tab_id}/back`, `/forward`, and `/reload` queue normal browser navigation controls; `GET /v1/tabs/active` returns the active tab if one exists. `DELETE /v1/tabs/{tab_id}` removes tab-scoped page context and cursor references, closes panes attached to that tab, queues browser close commands, and activates a remaining tab if the closed tab was active. The desktop profile stores browser tab URLs/titles in `tabs.json` and restores them on launch, but session IDs, grants, pane connections, and page-context snapshots are not restored. Owner clients can manage profile bookmarks through `GET/POST/DELETE /v1/bookmarks`; scoped pane grants cannot read or mutate bookmarks. Owner clients can also call `GET /v1/handoff` to export a non-secret continuity snapshot with current tabs, sessions, bookmarks, panes, page context, cursor refs, command results, and audit events.

The pane URL is configurable. Use whatever Codex login/session URL is appropriate for the environment instead of relying on a hardcoded default.

The desktop shell also supports local CLI panes for terminal agents. These are not `/v1/agent-panes` records and do not receive scoped bearer grants. The trusted chrome opens them as owner-local PTY processes, defaulting to `codex` or to `TELESCOPE_AGENT_CLI` when set. Type `claude` to run Claude Code instead. CLI panes dock on the same `left`, `right`, or `bottom` edges as web panes and have pane-local controls for `Pick`, `Add selection`, `Add page`, and `Close`. `Pick` queues the existing one-shot element picker against the active browser tab; after the user clicks a page component, `Add selection` writes a structured descriptor into the CLI session with the page URL, tab ID, element reference ID, selector, role, label, text preview, and bounds. `Add page` writes the latest active-tab page-context snapshot and visible interactive controls into the CLI session.

Opening a pane is not only stored as API state. `POST /v1/agent-panes` records the pane and queues a browser command for the attached tab. `POST /v1/scoped-agent-panes` is the owner-token convenience route for Codex: it derives or reuses an origin-scoped session for the active or requested tab, mints a short-lived grant bound to the pane URL origin, opens the pane, and returns the session, grant, pane, and connection. The runtime converts pane commands into an `open_agent_pane` adapter call. The desktop adapter docks a persistent native webview inside the main browser window at the requested edge, which gives Codex its own sign-in-capable browser surface without trying to embed the login page in an iframe. Left and right panes occupy side columns, bottom panes occupy a bottom strip, and multiple panes on the same edge split that edge.

Desktop browser tabs share the main browser profile context so signed-in user-site sessions are available across normal tabs. In the current desktop adapter, browser tabs are native webviews inside the main window; inactive tab webviews are hidden, and a trusted local chrome webview renders the tab strip, back/forward/reload buttons, address field, bookmark controls, saved-login controls, Codex web-pane controls, local CLI pane controls, and a recent audit-event strip above the page workspace. Chrome tab clicks, close/new-tab buttons, history/reload buttons, bookmark selections, and common window shortcuts queue the same control-plane tab commands exposed through the SDK. The native shortcuts are `Ctrl`/`Cmd+L` for the address field, `Ctrl`/`Cmd+T` for a new tab, `Ctrl`/`Cmd+W` to close the active tab, `Ctrl`/`Cmd+R` or `F5` to reload, and `Alt+Left`/`Alt+Right` for history. The bookmark controls save the active tab's current URL/title into `bookmarks.json` and navigate the active tab to selected bookmarks. The saved-login controls call owner-only tab credential routes: save derives the credential origin from the active tab URL, fill queues a browser command only after the saved credential origin matches the active tab URL, and password material stays in the browser runtime. The Codex web controls call the owner-only scoped-pane helper for the active tab, so opening a left/right/bottom web pane also creates the scoped session/grant the pane needs. The local CLI controls start PTY-backed `codex`, `claude`, or custom command panes without exposing the owner token to that process. The desktop chrome exposes the same credential, interaction, script, and read-only permission choices as the SDK/CLI instead of always opening the broad interactive grant. It also lists open Codex web panes with their permission summary and can stop a selected pane; for scoped panes, that stop path revokes the pane grant before queueing the browser close command. The desktop adapter restores prior browser tabs from `tabs.json`; a command-line URL is added as the active tab alongside restored tabs. Codex web panes share a separate agent profile context so Codex sign-in persists across left/right/bottom panes without mixing with the user's site cookies.

Scoped panes carry a separate `AgentPaneConnection`, not a token on plain pane state. The owner creates a short-lived grant, opens the pane with that grant, and the desktop adapter injects a frozen `window.__TELESCOPE_AGENT_CONNECTION` object into the pane webview. The same object is also available from the SDK's `open_scoped_agent_pane` result. It includes the local Telescope control URL, pane ID, tab ID, session ID, grant token, session policy snapshot, expiry, and relevant endpoint paths. The policy snapshot lets pane code see whether credentials, interactions, or custom scripts are allowed before trying an action. The pane grant is valid only while both the grant and its backing session are unexpired.

Rust SDK callers can use `ScopedAgentPaneOptions` with `open_codex_pane_with_options`, `open_scoped_agent_pane_for_tab_with_options`, or `open_codex_pane_for_active_tab_with_options` to choose the pane's credential, interaction, script, and TTL permissions explicitly. The simpler `open_codex_pane` helper requests credential and interaction access, while `open_read_only_codex_pane` requests only read-only page context, selector wait, extraction, and inspection access.

The SDK's `open_scoped_agent_pane` creates a grant bound to the pane URL's web origin for a caller-supplied session. `open_scoped_agent_pane_for_active_tab` and `open_scoped_agent_pane_for_tab` call the scoped-pane route and handle session creation/attachment for the owner. The local API responds to browser CORS preflights, including private-network preflight headers, so pane JavaScript can call the injected local control URL with its scoped bearer token. The preflight path is unauthenticated; all real API requests still require the owner token or a scoped grant, and origin-bound grant requests must present a matching `Origin` header. The Rust `ScopedAgentPane` helper sends that origin automatically for pane-bound grants.

The SDK also exposes a `BrowserTab` handle from `create_browser_tab`, `active_browser_tab`, `active_browser_tab_or_create`, or `browser_tab`. The same active-tab flow is available from the CLI with `telescope desktop open-codex <url> --position right|left|bottom`. By default it attaches to the active tab and requests credential and interaction access without custom scripts. Set `TELESCOPE_CODEX_URL` to prefill the desktop chrome Codex URL field or to run `telescope desktop open-codex` without a URL argument. Pass `--tab-id <tab-id>` to attach the Codex pane to a specific existing tab. Use `--read-only` for a context-only pane, or combine `--no-credentials`, `--no-interactions`, and `--allow-scripts` to tune the grant for narrower agent workflows. Owners can inspect and cut off individual pane grants with `telescope desktop grants` and `telescope desktop revoke-grant <grant-token>`, or use `telescope desktop stop-pane <pane-id>` to close a pane while revoking its scoped grant when present. Normal tab controls are available as `telescope desktop open <url>`, `telescope desktop new-tab [url]`, `telescope desktop activate-tab <tab-id>`, `telescope desktop close-tab <tab-id>`, `telescope desktop back`, `telescope desktop forward`, and `telescope desktop reload`. `telescope desktop doctor` prints the selected profile, control-file target, and platform webview prerequisites without reading or printing the owner token or connecting to the running browser. `telescope desktop context` prints the active tab's latest non-secret page context and visible interactive controls for terminal or SSH handoff; `telescope desktop refs` prints cursor-picked element references; `telescope desktop results` prints recent page-side command results. Pass `--tab-id` to `open`, `context`, `refs`, `results`, `back`, `forward`, or `reload` to target a specific tab, and pass `--json` for raw snapshots or queued-command records. Bookmark controls are available as `telescope desktop bookmarks list|save|open|delete`; pass `--tab-id` to `save` or `open` to target a specific tab, or pass `--new-tab` to `bookmarks open` to open the saved URL beside the current tab. Credential controls are available as `telescope desktop credentials list|save|fill|delete`; pass `--tab-id` to target a specific tab, and use `credentials fill --wait-seconds <n>` to wait for the browser-side fill result before returning. `telescope desktop handoff --json` exports the owner-visible continuity snapshot for reorientation after switching shells or SSH hosts, `--output <path>` writes that JSON to a private file, and `--restore <path>` applies its tab URLs and bookmarks to the current running browser. Rust callers can use SDK `handoff_snapshot` and `restore_handoff_snapshot` for the same workflow. These commands read the desktop `control.json` for the selected profile unless `--control-file` or `TELESCOPE_URL`/`TELESCOPE_TOKEN` are provided.

Owners can inspect the non-secret audit feed in the desktop chrome, with `GET /v1/audit-events`, SDK `list_audit_events`, or `telescope desktop audit`. Scoped pane grants cannot read that feed.

Pane lifecycle is explicit. `DELETE /v1/agent-panes/{pane_id}` queues a browser close command and can be called by the owner token or by the matching scoped grant. `DELETE /v1/agent-grants/{token}` is owner-only; it revokes the grant, removes pane connection descriptors tied to it, purges pending browser commands submitted by that grant, and queues close commands for those connected panes. `DELETE /v1/sessions/{session_id}` is the broader owner-only cutoff: it revokes all grants for the session, closes connected panes, detaches tabs from that session, and purges pending queued commands scoped to that session. Expired sessions and grants are cleaned up on the next control API request, so the browser runtime's normal command polling closes stale panes without a separate timer thread.

The context bridge has four pieces:

- Page context snapshots: title, URL, bounded text preview, optional selected element ID, and a capped list of visible interactive controls with selectors, roles, labels, input types, disabled state, and bounds. Scoped grants only see snapshots for their attached tab while the snapshot URL is still allowed by their session.
- Cursor element references: a one-shot picker records the clicked element's selector, role, label, text preview, and bounds. SDK `pick_element` and pane helper `pickElement` start the picker and wait for the next new reference on the attached tab, so an agent can ask the user to point at a component and immediately act on that exact reference. References are bound to the origin where they were captured; scoped agents stop seeing them after the tab moves to a disallowed origin, and actions reject a reference captured on a different origin than the current page.
- Fine-grained actions: SDK calls can inspect non-secret element metadata, scroll pages, scroll selectors or element references into view, click, double-click, drag between selectors or element references, hover, focus, type into, select dropdown options, toggle checkboxes or radio buttons, submit forms from, wait for, extract from, or otherwise act on selectors or element references.
- Page-side command results: login fill and action scripts post `telescope.command_result` over IPC with success/failure status and a bounded JSON message, letting the SDK wait for the actual browser outcome. Results carry non-secret command scope metadata and scoped grants only see results for commands issued by their own grant.

Page context snapshots update the tab's authoritative URL and title in the control plane. This keeps scoped agent permissions and credential-fill checks aligned with the page the browser actually loaded, including user-driven link clicks or redirects. Scoped pane navigation uses the same session/tab checks and can only navigate to a URL allowed by the attached session; read-only sessions without `allow_interactions` cannot navigate.

The desktop page-context capture intentionally does not read form field values. It captures labels, placeholders, roles, input types, selectors, and bounds so an agent can identify controls; the control plane also removes text from password inputs before storing snapshots.

Accepted fill-login and action commands carry the authorized target origin. The runtime embeds that origin into the page-side script and checks `window.location.origin` before interacting with the page, so commands fail closed if the tab moves to a different origin before execution.

Credential secrets are not part of this bridge. Login filling still resolves passwords only inside the browser runtime. Owner clients can use `POST /v1/tabs/{tab_id}/credentials`, `GET /v1/tabs/{tab_id}/credentials`, `POST /v1/tabs/{tab_id}/credentials/{credential_id}/fill`, and `DELETE /v1/tabs/{tab_id}/credentials/{credential_id}` for active-page credential workflows. The tab-scoped delete route only removes credentials whose origin matches the tab's current URL. Scoped agent grants can call `POST /v1/tabs/{tab_id}/login-options` to discover non-secret credential metadata for the current page only when their session allows credentials, then use the session-checked `/v1/tabs/{tab_id}/fill-login` route with the chosen credential ID.

Use an agent grant for the Codex pane instead of the owner token. The grant is tied to a session and optional tab IDs, so a Codex session can only see the page context and perform actions that match that scope. Session policy has separate bits for credentials, interactions, and custom scripts; scoped-pane creation creates a new session instead of reusing an existing tab session when the requested permissions differ.
