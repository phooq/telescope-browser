# Telescope Security Notes

Telescope treats the browser runtime as the only component allowed to resolve credential secrets.

The local control API is designed for automation and SDK access. It is not a password API:

- The owner bearer token can create sessions, manage credentials, and mint scoped agent grants.
- Agent grants are short-lived tokens scoped to a session and optional tab IDs. They are usable only while both the grant and its backing session are unexpired. They can read filtered page context, navigate within the session's allowed origins when interactions are allowed, and request allowed tab actions, but they cannot create sessions or manage credentials.
- Agent grants can also be scoped to client web origins. `open_scoped_agent_pane` and the `/v1/scoped-agent-panes` helper bind the grant to the Codex pane URL origin, and origin-bound grant requests are rejected unless they present a matching `Origin` header.
- The owner can revoke a whole agent session with `DELETE /v1/sessions/{id}`. Revocation removes the session, removes its grants, closes connected panes, detaches tabs from the session, and removes pending queued browser commands whose command scope was tied to that session.
- Expired sessions and grants are cleaned up before authenticated API handling. Expired sessions follow the same cleanup path as session revocation; expired grants close connected panes. The desktop runtime's regular command polling is enough to enqueue those pane closes after expiry.
- Browser tab creation and tab closing are owner-token operations. Closing a tab removes tab-scoped page context and cursor references, closes any attached agent panes, and leaves scoped grants unable to authorize that missing tab.
- Browser tab URLs accepted through the control plane are restricted to exact `about:blank` or `http`/`https` pages. Unsupported schemes such as `javascript:` and `data:` are rejected before tab state changes or browser commands are queued.
- The desktop address field can convert plain text into a search-engine URL. DuckDuckGo is the default, and `TELESCOPE_SEARCH_URL` can override the template. Search text is sent to that configured provider as normal browser navigation.
- An empty desktop profile starts on `about:blank`; Telescope does not contact a default website unless a startup URL is passed, saved tabs are restored, or the owner explicitly sets `TELESCOPE_HOME_URL` to an `http://`, `https://`, or `about:blank` page.
- Scoped Codex panes receive an `AgentPaneConnection` descriptor with a grant token and non-secret session policy snapshot for the attached tab/session. Plain pane listings do not include grant tokens; retrieving a pane connection requires the owner token or the matching grant.
- Grant listing is owner-only because it returns bearer-style grant tokens. Use it for local operator inspection and revocation, not as a pane-facing discovery API.
- Opening a scoped pane from desktop chrome or `open_scoped_agent_pane_for_active_tab` is still an owner-token operation. It derives the page session from the tab's current web origin, then gives the pane only the scoped grant token.
- The injected pane helpers `window.__TELESCOPE_AGENT` and optional `window.telescope` use only that scoped grant token. They can call the same page-context, element-reference, action, fill-login, command-result, and pane-close routes the grant could call manually.
- A scoped agent can close only the pane connected to its own grant. Grant revocation is owner-only, closes grant-connected panes, removes their connection descriptors, and purges pending browser commands that were queued by that grant but not yet polled by the runtime.
- The trusted desktop chrome lists open agent panes and can stop a selected pane. If the pane has a scoped grant connection, chrome stop revokes that grant so the token cannot continue to call the local API after the pane is closed, and any unpolled commands from that grant are discarded.
- The control API answers unauthenticated CORS preflight requests so pane web apps can use browser `fetch` against the local server. Authorization is still enforced on every non-preflight endpoint by bearer token, and origin-bound grants require a matching browser `Origin` header.
- The desktop browser and standalone `telescope serve` create or repair the active profile directory with `0700` permissions on Unix. They write `control.json` there so local tools can discover the live control server. It contains the owner token and is written or repaired with `0600` permissions on Unix; do not hand that file to Codex or another agent. Mint an agent grant instead.
- `telescope desktop doctor` validates control-file presence and shape without printing the owner token or connecting to the running browser. It is safe for local troubleshooting output, but paths can still reveal local profile names.
- Desktop and `telescope serve` profiles append owner-readable audit events to `audit.jsonl`. On Unix, new audit log files are created or repaired with `0600` permissions. The audit feed records session/grant creation, tab operations, credential metadata changes, credential-fill requests, agent-action summaries, pane lifecycle, and command-result status. The trusted desktop chrome also renders a compact recent-event strip from the same owner-only feed.
- `GET /v1/handoff` is owner-only. It exports non-secret continuity state for reorienting another local shell or SSH session: tabs, sessions, bookmarks, panes, page context, cursor refs, command results, and audit events. It does not include credential records or passwords, agent grant records, grant tokens, pending browser commands, or pane connection descriptors. `telescope desktop handoff --output <path>` writes that JSON with `0600` permissions on Unix. SDK `restore_handoff_snapshot` and `telescope desktop handoff --restore <path>` apply only tab URLs and bookmarks from that snapshot to the currently running browser.
- Desktop profiles store browser tab URLs and titles in `tabs.json` so human browsing state can be restored. That file does not include session IDs, agent grants, pane connection descriptors, page-context snapshots, cursor references, credential IDs, or credential secrets. On Unix, new tab snapshot files are written through a private `0600` temporary file before being renamed into place.
- Desktop and `telescope serve` profiles store owner-managed bookmarks in `bookmarks.json`. Scoped agent grants cannot read or mutate bookmarks, and new bookmark files are written with `0600` permissions on Unix.
- Desktop user tabs share one browser profile context for normal signed-in web sessions. Codex panes use a separate shared agent profile context, which keeps Codex login persistence separate from user-site cookies.
- Local CLI panes are owner-local PTY processes launched by the desktop shell, not origin-bound web panes and not scoped-grant API clients. They do not receive the owner token or grant tokens from Telescope. Their `Pick` and `Add selection` controls send only explicit, non-secret element metadata into the CLI session: page URL, tab ID, element reference ID, selector, role, label, text preview, and bounds. `Add page` sends the latest non-secret page-context snapshot.
- `POST /v1/credentials` stores a password. Saving another password for the same origin and username updates the existing credential ID and replaces the backing secret instead of creating duplicate records.
- `GET /v1/credentials` returns metadata only. On Unix, the credential metadata index is written through an owner-only `0600` temporary file before being renamed into place; raw password material stays in the configured `SecretStore`.
- `DELETE /v1/credentials/{id}` is owner-only and removes both credential metadata and the backing secret-store entry. `DELETE /v1/tabs/{tab_id}/credentials/{id}` also requires the owner token and only deletes when the credential origin matches the tab's current URL.
- `POST /v1/tabs/{id}/login-options` returns current-origin credential metadata only after session, tab, origin, and `allow_credentials` checks. It does not return password material.
- `POST /v1/tabs/{id}/fill-login` queues a browser fill command containing a credential ID and username only.
- Agent sessions have separate policy flags for credential use, page interactions, and custom scripts. Page-context reads, selector waits, text extraction, and non-secret element inspection can be allowed without allowing navigation or click/double-click/drag/hover/focus/type/select/toggle/scroll/key/submit actions.
- Scoped-pane creation reuses an existing tab session only when the requested credential, interaction, and script permissions exactly match that session. A read-only pane cannot inherit broader rights from an existing interactive tab session.
- Browser-observed page context updates the tab's authoritative URL and title. If the user clicks to a different origin, subsequent agent actions and credential fills are checked against that observed URL.
- Queued credential-fill and agent-action commands carry the origin that was authorized when the command was accepted. Runtime-injected scripts check `window.location.origin` before acting, so a page navigation between queueing and execution fails closed with an `origin_mismatch` command result.
- Password lookup is available only through in-process vault APIs intended for the browser runtime.
- `telescope-runtime` converts fill-login commands into webview JavaScript by resolving password material in process. Its debug output redacts sensitive scripts.
- Fill-login scripts report page-side success/failure over IPC with field-detection metadata only, such as whether a username or password field was found. They do not put raw passwords into command-result payloads.
- Fine-grained agent action scripts report page-side command results over the browser IPC bridge. Result messages are intended for action outcomes such as missing selectors and extracted page text; credential secrets are not placed in those payloads. Command results are tagged with the queued command's session/origin/grant scope, and scoped grants only receive results for commands issued by their own grant.
- Docked Codex panes receive page context, visible interactive-control metadata, and cursor-selected element references, not raw credential secrets. Page-context capture does not read form field values, and password-input text is dropped before storage. Scoped grants only receive page-context snapshots while the snapshot URL is allowed by their session and still matches the attached tab's current URL. Cursor element references are origin-bound when listed for scoped grants and when used for click/double-click/drag/hover/focus/type/select/toggle/scroll/submit/extract/inspect actions, so a stale reference from one origin cannot be replayed after the tab moves to another origin.
- Audit events are also non-secret: credential passwords, typed text payloads, custom script bodies, and bearer tokens are not serialized into the audit log.
- Arbitrary custom JavaScript actions require a session with `allow_scripts`; selector waits, text extraction, and element inspection require an origin-scoped session; click/double-click/drag/hover/focus/type/select/toggle/scroll/key/submit/picker actions also require `allow_interactions`. SDK `pick_element` and pane helper `pickElement` are wrappers around the same permissioned picker plus scoped element-reference reads.

Agent sessions use exact-origin allow lists. A session granted access to `https://example.com` is not granted access to `http://example.com`, another port, or another host.

The default `telescope-core` feature stores secrets in the OS keyring through the `keyring` crate. Tests and embedded use can inject a different `SecretStore`, such as the in-memory test store.
